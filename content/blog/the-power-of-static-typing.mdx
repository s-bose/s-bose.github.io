# The Power of Static Typing in Programming

There's a quiet confidence that comes with writing statically typed code. Before your program ever runs, the compiler
has already scanned every corner of your logic — silently catching the mistakes you didn't know you made.

## What Is Static Typing?

In a statically typed language, every variable has a **known type at compile time**. The compiler checks that
you're not passing a string where a number is expected, or calling a method that doesn't exist on a given object.

Languages like **TypeScript**, **Rust**, **Go**, and **Haskell** all enforce this discipline.

```typescript
function add(a: number, b: number): number {
  return a + b;
}

add(1, "two"); // ❌ Caught at compile time, not runtime
```

## Why It Matters

### 1. Errors Surface Early

The earlier a bug is caught, the cheaper it is to fix. A type error found at compile time costs seconds to fix.
The same error found in production can cost hours — or worse, data.

### 2. Self-Documenting Code

Types are a form of documentation that can't go stale. When you see a function signature like:

```typescript
function fetchUser(id: string): Promise<User>
```

You immediately know what goes in and what comes out. No need to dig through implementation details or hope the
comments are up to date.

### 3. Fearless Refactoring

Renaming a field in a large codebase is nerve-wracking in a dynamic language. In a statically typed one, the
compiler tells you every single callsite that broke. Refactoring becomes mechanical, not scary.

### 4. Better Tooling

Type information powers IDE features — autocompletion, inline documentation, go-to-definition, and rename
refactoring. The richer your types, the smarter your editor becomes.

## The Counterargument

Critics argue that static typing adds boilerplate and slows down prototyping. This used to be true of verbose
languages like Java circa 2005. Modern type systems with **inference** — like TypeScript, Kotlin, or Rust — have
largely eliminated that complaint.

```rust
let numbers = vec![1, 2, 3, 4, 5]; // Rust infers Vec<i32>
let doubled: Vec<_> = numbers.iter().map(|x| x * 2).collect();
```

You get the safety without writing types everywhere.

## Conclusion

Static typing is not a constraint — it's a **contract**. A contract between you and the machine, between you and
your future self, and between you and everyone else who reads your code. The small upfront investment in type
annotations pays compounding returns for the lifetime of a codebase.

Once you've worked in a well-typed system, going back feels like coding without a net.
